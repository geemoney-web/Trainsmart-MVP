Backend endpoints for qualification detail (QUAL-03) and unit detail with historical snapshots (QUAL-04, QUAL-05). Two new NestJS modules — `QualificationModule` and `UnitModule` — each with a controller/service pair following the existing `RtoModule` split pattern. Both register in `AppModule` and inherit the global `AccessTokenGuard` for authentication.

Purpose: Phase 3 frontend detail pages (Plan 04) consume these endpoints. Without them, the qualification and unit detail pages have nothing to render.
Output: Two endpoints, two controllers, two services, two Jest spec files covering happy-path + 404 + auth.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → API | Unauthenticated requests must not access qualification or unit data |
| RTO data isolation | Qualification/unit data is global (TGA-synced); no RTO scoping needed on these specific endpoints |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-02 | Spoofing | GET /qualifications/:qualId | mitigate | Global AccessTokenGuard from AuthModule (Phase 1) protects route by default; verify with auth-rejection test in qualification.controller.spec.ts |
| T-03-03 | Spoofing | GET /units/:unitId | mitigate | Same global guard inheritance; verified by unit.controller.spec.ts auth test |
| T-03-04 | Information Disclosure | qualification.findUnique include block | accept | Qualification and unit data is non-sensitive (publicly available from training.gov.au); no PII or RTO-specific data leaked |
| T-03-05 | Tampering | URL param `qualId` / `unitId` | mitigate | Use `findUnique({ where: { id } })` — Prisma validates UUID format and returns null for non-UUID strings, leading to 404 NotFoundException |

</threat_model>
